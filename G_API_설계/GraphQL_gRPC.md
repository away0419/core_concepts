# GraphQL · gRPC 도입 기준

REST 외에도 GraphQL과 gRPC가 널리 사용된다. 언제 어떤 기술을 선택할지 정리했다.

---

## 1. GraphQL
- 클라이언트가 필요한 필드만 선택.
- 단일 엔드포인트에서 다양한 리소스를 조합.
- Schema/Resolver 기반.
- **적합한 경우**: 모바일/웹에서 과다/과소 데이터 문제가 있을 때, BFF 계층.
- **주의**: 캐싱, 권한, N+1 문제에 대한 대비 필요.

## 2. gRPC
- HTTP/2 + Protobuf 기반 바이너리 프로토콜.
- 양방향 스트리밍, 낮은 지연, 엄격한 스키마.
- **적합한 경우**: 서비스 간 통신, 고성능 요구, 다국어 Stub 생성.
- **주의**: 브라우저 지원 한계, 로깅/디버깅 도구 필요.

---

## 3. 선택 체크리스트
- [ ] 사용 사례가 과다/과소 페이로드 문제인가? → GraphQL 고려.
- [ ] 내부 서비스 간 고성능 통신이 필요한가? → gRPC 고려.
- [ ] 팀이 Schema-First 개발 문화에 익숙한가?
- [ ] 모니터링/보안/권한 체계를 어떻게 적용할지 계획이 있는가?

---

## 4. 실습 아이디어
- **실습 1**: REST API를 GraphQL로 포팅하고, 쿼리/뮤테이션/서브스크립션을 작성.
- **실습 2**: gRPC 서비스와 클라이언트를 작성해 스트리밍 RPC를 구현.
- **실습 3**: GraphQL Federation 또는 gRPC-Gateway 등 하이브리드 아키텍처를 실험.
