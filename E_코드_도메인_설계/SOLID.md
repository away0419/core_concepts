# SOLID 5원칙 정리

SOLID는 객체지향 설계의 다섯 가지 핵심 원칙을 의미한다. 각 원칙을 이해하고 코드 예시와 함께 익혀두면 유지보수성이 크게 향상된다.

---

## 1. SRP (Single Responsibility Principle)
- 클래스/모듈은 단 하나의 변경 이유만 가져야 한다.
- 예: `UserService`가 인증/알림/통계까지 모두 담당하면 SRP 위반. 기능별 클래스로 나눈다.

## 2. OCP (Open/Closed Principle)
- 확장에는 열려 있고, 수정에는 닫혀 있어야 한다.
- 전략 패턴, 인터페이스 기반 확장으로 조건문 폭발을 막는다.

## 3. LSP (Liskov Substitution Principle)
- 하위 타입은 상위 타입을 대체할 수 있어야 한다.
- 메서드 시그니처/계약을 지키지 않으면 다형성이 깨진다.

## 4. ISP (Interface Segregation Principle)
- 클라이언트는 사용하지 않는 메서드에 의존하면 안 된다.
- 큰 인터페이스를 역할 단위로 나눈다.

## 5. DIP (Dependency Inversion Principle)
- 고수준 모듈은 저수준 구현에 의존하지 말고 추상화에 의존.
- 의존성 주입(DI), IoC 컨테이너가 DIP 구현을 도와준다.

---

## 6. 체크리스트
- [ ] 클래스/모듈이 담당하는 책임을 한 문장으로 설명할 수 있는가?
- [ ] 기능 확장을 위해 기존 코드를 수정하지 않고 새 클래스를 추가할 수 있는가?
- [ ] 상속 구조가 계약(LSP)을 위반하지 않는가?
- [ ] 인터페이스가 너무 크거나, 클라이언트가 필요 없는 메서드를 구현하고 있지 않은가?
- [ ] 추상화/인터페이스를 통해 의존성을 방향성 있게 관리하고 있는가?

---

## 7. 실습 아이디어
- **실습 1**: 냄새나는 클래스를 찾아 책임을 분리(SRP)하고, 테스트를 통해 리팩터링.
- **실습 2**: 조건문으로 분기하던 코드를 전략 패턴 + DIP로 개선.
- **실습 3**: 잘못된 상속 구조를 조합/위임으로 변경해 LSP를 만족시키기.
