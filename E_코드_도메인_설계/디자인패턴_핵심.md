# 대표 디자인 패턴 (Factory, Strategy, Template, Builder, Proxy, Observer)

각 패턴의 의도와 적용 예시를 간단히 정리해두면 필요할 때 빠르게 선택할 수 있다.

---

## 1. Factory Method / Abstract Factory
- **문제**: 객체 생성 로직이 곳곳에 흩어져 있을 때.
- **해결**: 생성 책임을 별도 Factory로 이동해 제품 계열을 숨긴다.
- **예시**: DB 커넥션, 메시지 파서 등.

## 2. Strategy
- **문제**: 알고리즘을 상황에 따라 바꿔야 할 때 if-else 남발.
- **해결**: 공통 인터페이스 + 전략 구현, 런타임에 교체.
- **예시**: 할인 정책, 결제 수단, 정렬 방식.

## 3. Template Method
- **문제**: 공통 골격은 같고 일부 단계만 다를 때.
- **해결**: 추상 클래스에 공통 흐름을 정의하고, 훅 메서드로 세부 구현을 위임.
- **예시**: 배치 작업, 훅이 있는 HTTP 필터.

## 4. Builder
- **문제**: 파라미터가 많은 객체 생성.
- **해결**: 단계적으로 값을 설정하고 마지막에 build().
- **예시**: 복합 설정 객체, SQL 쿼리 빌더.

## 5. Proxy
- **문제**: 원래 객체 앞단에 공통 부가기능(캐시, 보안 등)을 넣고 싶을 때.
- **해결**: 프록시가 원본과 동일한 인터페이스를 구현하고, 앞뒤로 로직 삽입.
- **예시**: RPC 클라이언트, AOP, Lazy 로딩.

## 6. Observer (Pub/Sub)
- **문제**: 상태 변화에 따라 여러 컴포넌트가 반응해야 할 때.
- **해결**: 옵저버 목록에 이벤트를 전달.
- **예시**: UI 이벤트, 도메인 이벤트, 메시지 브로커.
