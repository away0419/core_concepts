# Bounded Context 식별 가이드

Bounded Context는 DDD에서 모델의 경계를 정의하는 단위다. 각 컨텍스트는 독립된 모델, 용어, 데이터 저장소를 갖는다.

---

## 1. 어떻게 찾을까?
- 도메인 전문가와 대화해 “용어가 달라지는 지점”을 찾는다.
- 조직 구조, 팀 책임, 배포 주기 차이를 살펴본다.
- 모델 복잡도가 높아지는 영역을 분리한다.

---

## 2. Context Map 관계
- **Shared Kernel**: 일부 모델을 공유, 버전 관리 필요.
- **Customer/Supplier**: 한 컨텍스트가 다른 컨텍스트의 모델을 사용.
- **Anti-Corruption Layer**: 외부 시스템 모델을 보호하기 위한 번역 계층.
- **Published Language**: 이벤트/문서 등 명시적 계약.

---

## 3. 체크리스트
- [ ] 각 컨텍스트의 목표/용어/데이터 저장소가 독립적인가?
- [ ] 컨텍스트 간 통합 방식을 Context Map으로 명시했는가?
- [ ] 변경이 자주 겹치는 컨텍스트는 하나로 묶을지 검토했는가?
- [ ] 팀 구조와 컨텍스트 경계가 정렬되어 있는가?

---

## 4. 실습 아이디어
- **실습 1**: 현재 시스템의 주요 기능을 Context Map으로 표현.
- **실습 2**: Anti-Corruption Layer를 구현해 외부 API 모델을 내부 모델로 변환.
- **실습 3**: 이벤트 기반 통합으로 컨텍스트 간 결합을 줄이는 연습.
