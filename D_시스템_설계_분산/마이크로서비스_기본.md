# 마이크로서비스 기본과 단점

MSA(Microservices Architecture)는 기능을 작은 서비스 단위로 나누어 독립적으로 배포·운영할 수 있게 해준다. 하지만 복잡성과 운영 부담도 함께 따른다.

---

## 1. 핵심 개념
- 서비스는 독립적으로 배포/스케일/릴리즈 가능해야 한다.
- 각 서비스는 데이터베이스를 소유한다(공유 DB 지양).
- API 계약(REST/gRPC/Event)으로 통신하며, 느슨한 결합을 유지한다.

---

## 2. 장점
- 팀 규모가 커져도 병렬 개발 가능.
- 장애 격리: 특정 서비스 장애가 전체를 멈추지 않는다.
- 기술 스택 자유: 서비스별로 적합한 언어/DB 선택 가능.

## 3. 단점/도전 과제
- 분산 트랜잭션과 데이터 일관성 관리가 어려움.
- 네트워크 호출 증가 → 지연, 실패율 증가.
- 배포/모니터링/보안/테스트 자동화 수준이 높아야 함.
- 운영 비용 및 인프라 복잡도 증가.

---

## 4. 도입 체크포인트
- [ ] 도메인 별로 명확한 경계(팀/책임)를 나눌 수 있는가?
- [ ] CI/CD, Observability, Service Registry, API Gateway, Service Mesh 등 공통 인프라가 준비되어 있는가?
- [ ] 데이터 일관성을 보장할 패턴(SAGA, 이벤트 소싱)을 이해하고 있는가?
- [ ] 조직 문화가 DevOps/자율성을 지원하는가?

---

## 5. 실습 아이디어
- **실습 1**: 모놀리식 애플리케이션을 두 세 가지 도메인으로 나눠 API Gateway를 통해 호출.
- **실습 2**: 서비스별 독립 데이터베이스를 적용하고, 데이터 동기화 전략을 설계.
- **실습 3**: Chaos Engineering 도구로 특정 서비스 장애 시 전체 시스템이 어떻게 반응하는지 실험.
